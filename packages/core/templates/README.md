# Instruction File Templates (C3)

These templates generate the instruction files that auto-load when an AI agent starts a session. They're the routing layer — the mechanism that connects `.brand/` files to agent behavior.

## Template Inventory

| Template | Output File | Auto-loaded By | Purpose |
|----------|------------|----------------|---------|
| `CLAUDE.md.tmpl` | `CLAUDE.md` | Claude Code | Primary routing file |
| `AGENTS.md.tmpl` | `AGENTS.md` | Codex, Gemini CLI | Cross-tool equivalent |
| `cursorrules.tmpl` | `.cursorrules` | Cursor | Cursor equivalent |
| `copilot-instructions.md.tmpl` | `.github/copilot-instructions.md` | GitHub Copilot | Copilot equivalent |
| `impeccable.md.tmpl` | `.impeccable.md` | Impeccable skills | Brand personality data for design quality skills |

## Template Syntax

Templates use Mustache-style placeholders:

- `{{variable}}` — simple substitution from `.brandrc.yaml` or CLI flags
- `{{#section}}...{{/section}}` — conditional block (rendered if the value is truthy)
- `{{name}}` in routing rules is literal (not a variable) — it means the component name the agent is working on

## Design Decisions

### Why identical content across CLAUDE.md / AGENTS.md / .cursorrules / copilot-instructions.md?

The routing rules and enforcement rules must be identical across all tools. The brand routing mechanism works the same regardless of which agent reads it — the agent sees a task, matches a routing rule, loads the relevant `.brand/` file. Any divergence between tool-specific instruction files creates inconsistency.

`AGENTS.md` has one extra line explaining it's for non-Claude tools; otherwise the content is the same.

### Why the routing rules are this specific

Each routing rule maps one **task type** to one **file path**. This is intentional:

- "Any visual implementation → `.brand/tokens/`" is better than "read the brand files when relevant" because it removes agent judgment about what counts as relevant
- Explicit file paths let the agent find the file without searching
- "If a referenced file doesn't exist, proceed and flag the gap" means minimum-tier projects don't trigger errors — the agent just doesn't find `components/` files and moves on

### Why rules are numbered

Numbered rules signal enforcement priority. If rules conflict (rare), higher-numbered rules don't override lower-numbered ones — flag the conflict to the user instead.

### Why "When to Stop and Ask" exists

Without this section, agents will guess when they encounter ambiguity. Guessing produces plausible-looking but wrong output — brand-wrong colors, made-up component variants, invented tokens. The stop-and-ask list explicitly names the ambiguity cases where guessing is worse than pausing.

### Why `.impeccable.md` is a separate template

Impeccable skills read `.impeccable.md` from the project root via their own built-in protocol — not through CLAUDE.md routing. The file format is simpler (just brand personality data) because Impeccable doesn't need token values or workflow instructions. It needs the *feel* of the brand to make design judgment calls.

`.impeccable.md` is generated by `/brand-analyze` (C5) from `.brand/overview.md`. It can also be generated by the CLI from the overview file. The template defines the expected structure.

## Rendering

The CLI (`xd-brand init` / `xd-brand update`) renders templates:

1. Read `.brandrc.yaml` for project variables
2. For router templates (CLAUDE.md, AGENTS.md, etc.): substitute `client`, `project`, `deploy_platform`
3. For `.impeccable.md`: parse `.brand/overview.md`, extract fields, substitute
4. Write output files to project root (router files) or `.github/` (copilot)
5. On `update`: merge template changes, preserve any user customizations marked with `<!-- custom -->` comments

## Testing Brand Routing (Manual, Pre-CLI)

Before the CLI exists, you can manually test brand routing:

1. Copy a router template to a test project as `CLAUDE.md`
2. Replace `{{client}}` with the client name, `{{deploy_platform}}` with `Netlify`
3. Remove the `{{#project}}` conditional (or fill it in)
4. Create a `.brand/` directory with at minimum: `overview.md`, `voice.md`, `tokens/colors.md`, `tokens/typography.md`, `tokens/spacing.md`
5. Start a Claude Code session — verify that the agent reads `overview.md` at session start
6. Ask the agent to "build a hero section" — verify it loads `composition/page-types.md` and `tokens/` files
7. Ask the agent to "write a button label" — verify it loads `voice.md`
